# 扫描策略清理 - 对齐阶段文档

## 原始需求

用户要求按照6A工作流方式，删除所有与 `'metadata' | 'media' | 'full'` 扫描策略重构相关的无效代码，只保留 `'legacy'` 和 `'unified'` 两种策略。

## 项目背景分析

### 扫描策略演进历史
1. **初始状态**: 只有传统的 `legacy` 策略
2. **第一次重构**: 增加了 `'metadata' | 'media' | 'full'` 三种策略
3. **第二次重构**: 发现问题，引入 `unified` 策略
4. **当前需求**: 移除中间重构的三种策略，只保留 `legacy` 和 `unified`

### 当前架构现状

从 `ScanOrchestrator.ts` 分析，当前系统包含以下策略：
- ✅ `legacy` - 需要保留
- ❌ `metadata` - 需要移除
- ❌ `media` - 需要移除  
- ❌ `full` - 需要移除
- ✅ `unified` - 需要保留

## 影响范围分析

### 1. 类型定义层面
- `ScanStrategyType` 类型定义需要更新
- 相关接口和枚举需要调整

### 2. 策略实现层面
- `MetadataScanStrategy` 类文件需要删除
- `MediaScanStrategy` 类文件需要删除
- `FullScanStrategy` 类文件需要删除
- `ScanOrchestrator` 中的策略注册需要清理

### 3. 上游调用层面
- API路由中的策略选择逻辑
- 前端界面中的策略选项
- 配置文件中的默认策略设置

### 4. 下游依赖层面
- 测试文件中的策略测试
- 文档中的策略说明
- 示例代码中的策略使用

## 边界确认

### 包含范围
- 删除 `metadata`、`media`、`full` 三种策略的所有实现代码
- 更新类型定义，移除这三种策略类型
- 清理 `ScanOrchestrator` 中相关的注册和逻辑
- 更新API接口，移除对这三种策略的支持
- 更新前端界面，移除相关选项
- 清理相关测试和文档

### 排除范围
- 不影响 `legacy` 和 `unified` 策略的正常功能
- 不修改核心扫描逻辑，只移除策略层面的代码
- 保持现有API的向后兼容性（对于支持的策略）

## 需求理解

### 核心目标
1. **代码简化**: 移除不需要的策略实现，简化代码结构
2. **架构清理**: 消除中间重构带来的技术债务
3. **维护性提升**: 减少策略数量，降低维护复杂度
4. **性能优化**: 减少不必要的策略选择和判断逻辑

### 技术约束
1. **保持功能完整性**: `legacy` 和 `unified` 策略必须正常工作
2. **向后兼容**: 现有使用这两种策略的代码不能受影响
3. **渐进式清理**: 按依赖关系逐步清理，避免破坏性变更
4. **测试覆盖**: 确保清理后的代码有充分的测试覆盖

1. **默认策略**: 清理后的默认策略应该是 `legacy` 还是 `unified`？

3. **配置迁移**: 现有配置文件中指定了被删除策略的情况如何处理？
4. **错误处理**: 当用户尝试使用被删除的策略时，应该返回什么错误信息？

3. **日志记录**: 是否需要在日志中记录策略清理的相关信息？

### 低优先级问题

2. **监控指标**: 现有的监控指标是否需要调整？
