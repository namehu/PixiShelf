# 媒体类型后缀匹配技术债

## 背景

本次媒体派生系统标签只解决了 `image`、`video`、`webp` 标签的自动同步问题，没有把作品列表、播放器、导入流程里所有“通过文件后缀判断媒体类型”的旧逻辑一起重构。

目前项目里仍然存在一些运行时后缀判断：

- 作品列表筛选 `mediaType = image | video` 仍在服务层通过 `Image.path` 后缀匹配。
- `transformImages` 仍根据路径后缀输出前端用的 `mediaType`。
- 上传、替换、扫描等流程仍通过扩展名判断文件是否属于支持的媒体格式。
- 前端渲染仍会根据媒体路径判断走图片、视频、WebP 动图或 APNG 逻辑。

这些逻辑短期保留是为了避免把“系统标签同步”与“媒体模型重构”混在同一次变更里，降低生产风险。

## 问题

按后缀判断媒体类型有几个明显缺点：

1. 同一套规则散落在服务层、导入层、前端渲染层，容易出现不一致。
2. 查询时用 `LIKE / endsWith` 做后缀匹配，数据量变大后不如结构化字段稳定。
3. `Image` 表名已经和真实语义不匹配，实际承载的是图片、视频、WebP、APNG 等媒体条目。
4. 系统标签现在是派生结果，但媒体筛选本身仍不依赖结构化媒体类型。
5. 如果未来支持更多格式，需要同时改多个地方。

## TODO

### 1. 引入结构化媒体类型字段

给当前 `Image` 表增加媒体类型字段，字段名可先用：

```prisma
mediaType String @default("image")
```

或者更严格地使用 enum：

```prisma
enum MediaType {
  IMAGE
  VIDEO
}
```

短期建议先用 `String`，因为当前项目还存在 APNG、WebP 动图等细分格式，后续可以再拆 `mediaKind` / `format`。

### 2. 引入媒体格式字段

为了避免 `webp` 这类格式继续靠路径判断，可以额外存：

```prisma
format String?
```

示例：

```text
mediaType = image, format = webp
mediaType = image, format = jpg
mediaType = video, format = mp4
mediaType = video, format = webm
```

这样后续 `webp` 标签、视频筛选、格式筛选都可以从结构化字段读取。

### 3. 集中媒体识别规则

保留一个统一函数，例如：

```ts
detectMediaMetadata(path: string): {
  mediaType: 'image' | 'video'
  format: string
}
```

所有上传、扫描、替换、批量导入都调用它，不再各自写扩展名判断。

### 4. 回填历史媒体字段

新增“数据修正”任务：

- 扫描所有 `Image.path`。
- 根据统一识别函数回填 `mediaType` 和 `format`。
- 分批处理，默认每批 1000 条。
- 输出处理数量、跳过数量、失败数量。

### 5. 查询逐步切换到结构化字段

在历史数据回填完成后，把作品列表筛选从：

```sql
LOWER(i.path) LIKE '%.mp4'
```

逐步切换为：

```sql
i."mediaType" = 'video'
```

`webp` 相关逻辑也从 `path endsWith ".webp"` 切换为：

```sql
i."format" = 'webp'
```

### 6. 再考虑表命名重构

`Image` 当前实际是媒体表。等字段和调用点稳定后，再评估是否改名为 `Media`。

不建议和上面的字段迁移放在同一次做，因为表名重构会影响 Prisma relation、DTO、前端命名、API 命名和大量 UI 文案。

## 验收标准

- 新增/替换/扫描媒体时，数据库会写入结构化 `mediaType` 和 `format`。
- 历史数据可以通过后台“数据修正”完成回填。
- 作品列表的图片/视频筛选不再依赖 `Image.path` 后缀匹配。
- `webp` 系统标签同步可以基于 `format = webp`。
- 前端渲染仍兼容旧数据，在字段缺失时可以临时 fallback 到路径后缀判断。

## 当前取舍

本次不做这件事是合理的：媒体派生标签是小闭环，风险集中在标签关系；结构化媒体类型是更底层的数据模型重构，需要单独设计、迁移和验证。

后续做这个 TODO 时，应先补字段和回填任务，再逐步切查询逻辑，最后再考虑 `Image` 到 `Media` 的命名重构。
